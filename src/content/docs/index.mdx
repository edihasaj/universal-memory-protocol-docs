## The third interoperability layer

Agents can already call tools (**MCP**) and talk to each other (**A2A**). What they
can't do is **carry memory across sessions, agents, and vendors**. Every harness
reinvents memory privately and non-portably. UMP fixes that.

In simple terms: you already have useful memories scattered across agent files,
Claude/Codex project notes, Recall exports, Obsidian folders, Postgres, Redis,
SQLite, and vector databases. UMP turns those into one portable memory shape and
one small operation set, so new agents and new stores can extend the same memory
instead of starting over.

<FeatureGrid>
  <Feature title="Tools - MCP" icon="mcp">
    Model Context Protocol standardized how agents call functions and read resources.
  </Feature>
  <Feature title="Coordination - A2A" icon="a2a">
    Agent2Agent standardized how agents discover and invoke one another.
  </Feature>
  <Feature title="Memory - UMP" icon="memory">
    Universal Memory Protocol standardizes how agents **remember** - portable knowledge across sessions, agents, and vendors.
  </Feature>
</FeatureGrid>

## Implement it

Use the MCP server when you want agent memory immediately. Use the TypeScript SDK
when you are building a memory-aware app or agent runtime. Use HTTP when the
client is Python, Go, Swift, a browser, or anything else that can send JSON.

```jsonc
// MCP host config: Claude Code, Codex, Cursor, or any MCP client.
{
  "mcpServers": {
    "ump": {
      "command": "npx",
      "args": ["-y", "@universalmemoryprotocol/core", "ump-memory"]
    }
  }
}
```

```ts
import {
  JsonFileStore,
  UmpServer,
  generateKeyPair,
} from "@universalmemoryprotocol/core";

const key = generateKeyPair();
const store = await JsonFileStore.open(".ump/memory.ump.json");
const ump = new UmpServer({
  name: "my-agent",
  version: "1.0.0",
  conformance: "L2",
  store,
  key,
});

await ump.remember({
  kind: "procedural",
  body: { text: "Use pnpm for this repository." },
  scope: { owner: key.did, project: "github.com/acme/app", visibility: "private" },
  provenance: { actor: key.did, actor_kind: "user", method: "user_correction" },
});

const memories = await ump.recall({
  query: "package manager",
  scope: { owner: key.did, project: "github.com/acme/app" },
});
```

```bash
# Any language: expose JSON over HTTP.
UMP_HTTP=4000 npx -y @universalmemoryprotocol/core ump-memory
```

```py
import requests

base = "http://localhost:4000"
owner = requests.get(f"{base}/.well-known/ump.json").json()["owner"]

requests.post(f"{base}/ump/remember", json={
    "kind": "semantic",
    "body": {"text": "User prefers concise release notes."},
    "scope": {"owner": owner, "project": "github.com/acme/app", "visibility": "private"},
    "provenance": {"actor": owner, "actor_kind": "user", "method": "user_correction"},
}).raise_for_status()

hits = requests.post(f"{base}/ump/recall", json={
    "query": "release note preference",
    "scope": {"owner": owner, "project": "github.com/acme/app"},
}).json()["results"]
```

## Why it's shaped like this

UMP is **not a new transport protocol**. It is an application-level memory
protocol that rides existing transports. The lesson of MCP is *minimal primitives
+ existing rails + great SDKs + neutral governance*.

<FeatureGrid>
  <Feature title="Rides MCP" icon="rides">
    The primary binding is an MCP profile (`ump.*` tools). Any MCP host - Claude
    Code, Codex - can use UMP today, with zero new transport.
  </Feature>
  <Feature title="Six operations" icon="ops">
    `capabilities · recall · remember · revise · forget · get` (+ optional
    `feedback`/`subscribe`). A conforming client is ~100 lines.
  </Feature>
  <Feature title="One portable record" icon="record">
    Typed, scoped, bi-temporal, signed JSON. Reuses W3C PROV + DID - no new vocabulary.
  </Feature>
  <Feature title="Adopt incrementally" icon="incremental">
    Four conformance levels. Ship an `*.ump.json` export (L0) today; wire the full
    runtime (L3) when ready.
  </Feature>
</FeatureGrid>

## Any store can serve it

UMP is a protocol surface, not a database bet. [`@universalmemoryprotocol/core`](https://www.npmjs.com/package/@universalmemoryprotocol/core) ships practical
implementations for local files, SQL databases, Redis, Recall, and vector engines
so teams can adopt UMP where they already store memory.

<FeatureGrid>
  <Feature title="Files first" icon="files">
    `JsonFileStore` writes portable `memory.ump.json`; `MarkdownDirectoryStore`
    writes human-editable `*.ump.md` records for repos and Obsidian-style vaults.
  </Feature>
  <Feature title="SQL + cache" icon="sql">
    `PostgresStore`, `SqliteStore`, and `RedisStore` accept your existing client.
    No native database driver is bundled into the core package.
  </Feature>
  <Feature title="Vector databases" icon="vector">
    `VectorStore` plus `QdrantStore`, `PineconeStore`, and `WeaviateStore` wrappers
    let embedding-backed engines keep their own retrieval quality.
  </Feature>
  <Feature title="Recall engine adapter" icon="recall">
    Recall is one implementation: a richer memory engine behind the same UMP
    interface, not a dependency or lock-in.
  </Feature>
</FeatureGrid>

## Bring existing memory with you

People already have memory scattered through `AGENTS.md`, `CLAUDE.md`, Recall
exports, Obsidian notes, and project Markdown. UMP keeps those formats separate,
but ships import bridges that turn them into portable UMP drafts with provenance.

<FeatureGrid>
  <Feature title="Agent files" icon="agentfiles">
    `ump-import` reads `AGENTS.md` (the open standard) and `CLAUDE.md` as
    procedural candidate memories, preserving the source path in provenance.
  </Feature>
  <Feature title="Vaults and folders" icon="vaults">
    Point it at Markdown directories or Obsidian-style vaults to bootstrap a
    local UMP store without changing your notes.
  </Feature>
  <Feature title="Bridge, not lock-in" icon="bridge">
    Importers are adoption helpers. UMP remains the common record and runtime
    interface after the data is translated.
  </Feature>
</FeatureGrid>

## What's standardized - and what isn't

UMP standardizes **structure, provenance, access, and trust**. It deliberately
leaves **intelligence** - extraction, ranking, decay, consolidation - to the
engine underneath, so implementations compete on quality while staying interoperable.

<FeatureGrid>
  <Feature title="Bi-temporal, supersede-never-delete" icon="temporal">
    Valid-time + transaction-time. A fact that changes is closed and linked to its
    successor - not overwritten. The credible answer to staleness.
  </Feature>
  <Feature title="User-owned & signed" icon="signed">
    The operator - not the model vendor - holds the key. Records are content-addressed
    and signed with a DID. Portable, verifiable, no lock-in.
  </Feature>
  <Feature title="Injection-resistant by mandate" icon="injection">
    Memory is attacker-controllable input. The spec *requires* a verify, filter,
    frame rehydration pipeline. Never string-interpolated into the prompt.
  </Feature>
  <Feature title="Consent in the record" icon="consent">
    Retention, exportability, and redaction travel *with* the data and are enforced
    at every boundary.
  </Feature>
</FeatureGrid>
